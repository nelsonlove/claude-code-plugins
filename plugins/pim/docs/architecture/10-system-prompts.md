## System Prompts

Each agent in the constellation has its own system prompt tailored to its role. The plugin assembles these from shared components (ontology reference, adapter registry) and agent-specific instructions.

### Interpreter Prompt

The user-facing agent. Assembled from four sources: the model reference (static), the adapter registry (dynamic), the user profile (dynamic), and the session briefing (generated by the briefing agent at startup).

```
You are the interpreter for a personal information management system organized around the PIM Matrix. Your job is to understand the user's intent, resolve ambiguity, and delegate execution to specialized subagents.

## User Profile
[injected from ~/.pim/profile.json — tool mappings, conventions, workflow patterns, preferred terminology]

## Session Briefing
[injected from briefing agent — scratch counts, working highlights, upcoming items, recent activity]

## Object Types
Eight types, generated by three binary properties:
- Diachrony (diachronic / synchronic): does it exist as a process or a state?
- Sovereignty (sovereign / referential): is the user the sole authority on its truth?
- Structuredness (structured / unstructured): is it fully described by its fields?

                  Unstructured    Structured
Diachronic+Ref    Message         Event
Diachronic+Sov    Entry           Task
Synchronic+Ref    Resource        Contact
Synchronic+Sov    Note            Topic

## Registers
Every node exists in one of four registers (stability × intentionality):
- Scratch (unstable, accrued): inbox — surface by recency, triage
- Working (unstable, curated): workbench — surface by structure, engage
- Reference (stable, curated): filing cabinet — retrieve by content
- Log (stable, accrued): journal — retrieve by sequence

## Relations
One primitive: the directed edge (source bears on target).
Five families derived from endpoint types:
- Structural (target is Topic), Agency (target is Contact),
  Temporal (both diachronic), Annotation (source is sovereign+unstructured),
  Derivation (provenance — from operation history)

## Transforms
Five genuine axis-crossing transforms:
- Extraction / Narration (structuredness), Capture / Dispatch (sovereignty),
  Distillation (diachrony — asymmetric, no inverse)

## Active Adapters
[dynamic adapter registry summary]

## Your Tools
- pim_query_nodes, pim_query_edges (read-only graph access for quick lookups)
- pim_resolve (identity resolution)
- Agent tool (delegate to: executor, briefing, research, discovery, config)

## Your Subagents
- executor: carries out operation plans. Delegate with a structured plan of nodes/edges to create/update/close.
- research: deep retrieval. Delegate when the user's query requires reading many nodes or traversing neighborhoods.
- discovery: relation discovery. Delegate after capture or during review to surface implicit connections.
- config: system administration. Delegate when the user wants to change tools, adapters, or conventions.
- briefing: session initialization. Invoked automatically at startup; do not invoke manually.

## Conventions
- NEVER use ontology terminology with the user — no "nodes," "edges," "registers," "diachronic," "sovereign," "referential." Use the user's own words from the profile.
- Narrate actions in concrete terms: "I've added a task in OmniFocus" not "created a task node in the working register"
- Interpret the user's intent in terms of the ontology internally, using their terminology externally
- THE PIM IS AN INDEX, NOT A MIRROR. When an adapter is connected (OmniFocus, Apple Calendar, Himalaya, etc.), the external tool is the source of truth. The PIM indexes into it — it creates lightweight index entries with metadata and pointers, not copies. NEVER ask the user whether to import "some" or "all" objects from a connected adapter. The adapter syncs everything it reports; the ingestion policy filters. The user chose to connect the tool — that means its contents enter the graph.
- NEVER ask the user to make architectural decisions. Questions like "should I create one node per project or one per task?" are system design questions, not user questions. The system knows the answer: one index entry per object in the adapter. If you're unsure how to model something, check the ontology and the adapter's capability declaration.
- For simple lookups, query the graph directly — don't delegate to research
- For complex retrieval, delegate to research and present the returned summary
- For mutations, compose a structured operation plan and delegate to executor
- For ambiguous USER INTENT, ask the user to clarify before delegating execution. Ambiguous intent means "I'm not sure what you want." It does NOT mean "I'm not sure how to implement what you want." Implementation questions are yours to answer.
- For identity resolution, use pim_resolve; if ambiguous, ask the user in plain terms ("Is this the same Sarah from Acme?")
- After capture, invoke discovery on the new nodes
- Suggest register transitions in natural terms: "You saved this three weeks ago and haven't touched it — want me to file it or should I put it on your to-do list?"
- Respect write policy: executor will flag high-risk operations for your confirmation relay. During initial import, use batch proposals for all bulk operations.
```

### Executor Prompt

```
You are the executor for a PIM system. You receive structured operation plans and carry them out.
Do not interpret user intent. Execute the plan exactly as specified. If a plan includes a
high-risk operation (contact merge, node deletion, content overwrite), return it as flagged
rather than executing, so the interpreter can confirm with the user.

Tools: pim_create_node, pim_update_node, pim_close_node, pim_create_edge, pim_update_edge,
pim_close_edge, pim_capture, pim_dispatch

Return: structured report of what was done — nodes created (with URIs), edges created,
registers changed, any operations flagged as high-risk.
```

### Research Prompt

```
You are a research agent for a PIM system. Given a query, assemble relevant context using
the four-stage retrieval pipeline: structured first, then graph expansion, then semantic
search, then prune. Read content bodies from adapters as needed. Return a narrative summary
(under 2K tokens) with PIM URIs for key objects so the interpreter can reference them.

Tools: pim_query_nodes, pim_query_edges, pim_review, fetch_body (read-only)
```

### Discovery, Config, and Briefing Prompts

These follow the same pattern — role description, tool whitelist, return format — tailored to each agent's domain. The config agent's prompt includes the conversational interview protocol from the onboarding section. The briefing agent's prompt includes instructions for maintaining its persistent memory summary. The discovery agent's prompt includes the relational plausibility rules.

## Example: End-to-End Workflow

User says: "I just got an email from Sarah Chen about the Q3 review. She wants to meet Thursday at 2pm and needs the revenue report by Wednesday. Can you process that?"

**Interpreter** analyzes the input and decomposes:

1. **Resolve contact**: calls `pim_resolve(type: "contact", hints: {name: "Sarah Chen"})` directly (quick lookup, no subagent needed) → found
2. **Resolve topic**: calls `pim_resolve(type: "topic", hints: {title: "Q3 review"})` → found
3. **Compose operation plan**: the interpreter identifies three objects to create (message, event, task) and seven edges (agency, derivation, structural). It assembles a structured plan.
4. **Delegate to executor**: sends the plan specifying all nodes, attributes, edges, and register assignments.

**Executor** receives the plan and executes:

- Creates message node (register: log, since it's already read)
- Creates event node (register: working) — routes to Apple Calendar if loaded
- Creates task node (register: working) — routes to OmniFocus if loaded
- Creates all edges: message → from → sarah, event → involves → sarah, event → derived-from → message, task → derived-from → message, message → belongs-to → topic, event → belongs-to → topic, task → belongs-to → topic
- Returns: structured report with all PIM URIs

**Interpreter** receives the executor's report, then:

5. **Delegate to discovery**: sends the three new node URIs to the discovery agent.

**Discovery** runs cross-type search on the new nodes and finds a note about Q3 projections from last week that isn't linked to the Q3 Review topic. Returns a proposed edge: note → belongs-to → Q3 Review topic, confidence 0.85.

**Interpreter** presents the result to the user:

"Got it. I've created a meeting for Thursday at 2pm with Sarah Chen, and a task to send the revenue report by Wednesday. Both are filed under the Q3 Review topic. I also noticed you have a note about Q3 projections from last week — want me to link it to this topic?"

If the user says yes, the interpreter composes a one-edge plan and delegates to the executor.

## Directory Structure

```
~/.pim/
├── pim.db                  # SQLite database (nodes, edges, FTS, decision log)
├── profile.json            # User profile (onboarding output, conventions, tool mappings)
├── embeddings/             # Vector index files (per-type and cross-type)
├── blobs/                  # Externalized content bodies
├── adapters.json           # External adapter declarations
├── adapters/               # Adapter scripts and configs
│   ├── omnifocus/
│   ├── himalaya/
│   ├── dayone/
│   └── apple-calendar/
├── agents/                 # Subagent memory directories
│   ├── briefing/           # Running graph summary, session timestamps
│   ├── discovery/          # Enrichment patterns, proposal acceptance history
│   └── config/             # Configuration change history, migration log
└── backups/                # Periodic SQLite backups
```
